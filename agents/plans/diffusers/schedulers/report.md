## 2. Kernel suitability frame

Most Diffusers scheduler `step` methods are not compute-heavy compared with the
denoiser, but they are bandwidth-bound full-latent pointwise passes. A custom
fused CUDA kernel is useful when it can remove several launches around the
denoiser loop: `scale_model_input`, classifier-free guidance arithmetic,
prediction conversion, scheduler step, optional `add_noise`, and optional mask
mixing.

Use this report's suitability labels as follows:

| Label | Meaning |
| --- | --- |
| High | Deterministic or externally-noised pointwise latent update. Good first custom fused CUDA kernel candidate. |
| Medium | Pointwise update is worthwhile, but host-visible history, warmup, stochastic noise, masks, or variant flags should be staged first. |
| Low | CUDA kernels may help pieces, but the scheduler is dominated by sampling, sorting, categorical logits, reductions, interpolation, or rare pipeline semantics. |
| Defer | Deprecated, non-CUDA implementation mirror, or not part of first diffusion-loop parity. |

Important constraint: a "custom fused scheduler kernel" should still be admitted
as a scheduler-family-specific provider or generated fused op, with explicit
tables and coefficients in artifacts. It should not hide mutable Python
scheduler state inside generated CUDA.

## 3. General scheduler runtime plan

### Host-owned schedule generation

Keep `set_timesteps(...)` and scheduler admission on the host/runtime side for
first integration. It performs scheduler-specific validation, custom
timesteps/sigmas checks, dynamic shift logic, Karras/exponential/beta schedule
conversion, strength slicing, begin-index behavior, and multistep warmup
bookkeeping. These choices are small compared with denoiser work and are
important parity state.

Artifacts should record a scheduler plan like:

```text
scheduler_family
source_config fields used for schedule generation
num_train_timesteps
num_inference_steps
timestep_spacing / custom_timesteps / custom_sigmas admission
prediction_type and model_output_semantics
tables: timesteps, sigmas, alphas_cumprod, or precomputed coeffs as applicable
state contract: step_index, begin_index, history slots, lower_order counters
```

### Kernel ABI decision: pass exact coefficients first

For first fused CUDA kernels, pass the exact per-step values required by that
kernel, not the whole `timesteps`/`sigmas` table plus an index.

Recommended first ABI shape:

```text
kernel(sample, model_output, optional_guidance_parts, optional_noise,
       output, n_elements, SchedulerStepCoefficients coeffs,
       optional_history_buffers, optional_mask_or_original)
```

Where `SchedulerStepCoefficients` is scheduler-family-specific or a small tagged
union generated by the compiler. Examples:

```text
FlowMatch Euler:
  sigma, sigma_next, dt, stochastic flag

EulerDiscrete:
  sigma, sigma_hat, sigma_next, inv_sqrt_sigma2_plus_1, prediction_type

DDIM:
  sqrt_alpha_t, sqrt_beta_t, sqrt_alpha_prev,
  direction_scale, eta_std, prediction_type, clip/threshold policy

LCM:
  sqrt_alpha_t, sqrt_beta_t, c_skip, c_out,
  sqrt_alpha_prev, sqrt_beta_prev, is_last_step
```

Rationale:

- The host already owns schedule validation and the current step index.
- The exact coefficient pack makes the kernel independent of Diffusers' many
  table layouts and duplicate-timestep lookup rules.
- Custom sigmas/timesteps, strength slicing, and `begin_index` become host
  concerns instead of device-side search concerns.
- Per-family coefficient packs make parity tests direct: compare host-generated
  coeffs and one kernel step against Diffusers.
- It avoids device table reads and branches in bandwidth-bound kernels.
- It keeps multistep history explicit through separate pointers or tensor args.

Keep full tables artifact-visible anyway. They are needed for auditability,
validation, schedule replay, host-side coefficient selection, and future CUDA
graph or persistent-loop execution.

### When table plus index is acceptable

Passing schedule tables plus an index can be added later for a generated loop
kernel or CUDA graph path where the GPU owns repeated denoising-loop dispatch.
Even then, prefer a compact per-step coefficient table over raw Diffusers-style
`timesteps`/`sigmas` tables. A raw table+index ABI should be limited to debug
fallbacks or simple `add_noise` kernels where the table is the natural input.

### Per-token timestep exception

Flow-match per-token timesteps are not a scalar-step ABI. For those branches,
pass the exact per-token current/next sigma tensors or a compact per-token
coefficient tensor. Do not ask a fused kernel to recover them by searching the
global schedule table unless that search itself is the tested operation.

### Stochastic branches

Do not fuse random-number generation into first scheduler kernels. First parity
should accept an explicit `noise` tensor from the runtime, then compute the
deterministic combination in CUDA. Later provider work can decide whether a
DinoML RNG stream belongs in scheduler kernels.

## 5. Flax scheduler handling

Diffusers also exports Flax variants for DDIM, DDPM, DPM-Solver multistep,
Euler, Karras VE, LMS, PNDM, and SDE VE. For DinoML CUDA planning they should
not become independent scheduler targets. Use the PyTorch scheduler semantics
above as the parity source, then treat Flax files as non-CUDA mirrors unless a
future JAX import path explicitly asks for them.

## 6. Recommended first fused CUDA scheduler slices

1. `FlowMatchEulerDiscreteScheduler`, deterministic scalar-sigma path.
   Covers SD3/Flux first parity and has the cleanest step:
   `prev = sample + (sigma_next - sigma) * model_output`.

2. `EulerDiscreteScheduler`, no-churn path.
   Covers SDXL base/refiner and many SD scheduler swaps. Fuse
   `scale_model_input`, prediction conversion, optional CFG arithmetic, and
   Euler update where the denoiser staging allows it.

3. `DDIMScheduler`, deterministic eta=0 alpha-product path.
   Covers SD2 v-pred and a simple SD baseline slice. Keep stochastic eta and
   dynamic thresholding separate.

4. `CogVideoXDDIMScheduler`, v-pred video path.
   Same general alpha-product kernel shape as DDIM, but coefficient generation
   and SNR shift should be separate scheduler-family metadata.

5. `LCMScheduler` or `TCDScheduler`, deterministic or externally-noised.
   Good distilled-model follow-up once DDIM coefficients and explicit-noise ABI
   are in place.

6. `UniPCMultistepScheduler`, fixed Wan flow-prediction `bh2` order-2 path.
   Important for Wan, but it needs scheduler history buffers and coefficient
   tables before it should be fused.

## 7. Implementation notes for DinoML admission

- Do not admit a generic "DiffusersScheduler" op. Admit by scheduler family,
  prediction type, stochastic/noise mode, and state contract.
- Keep `scale_model_input` separate in the schema even when a fused kernel later
  combines it with CFG or denoiser input staging.
- Store `init_noise_sigma`, timesteps, sigmas, alpha products, and selected
  coefficient table in a manifest-visible schedule plan.
- Store `begin_index` and `step_index` semantics as runtime state, not hidden
  Python object mutation.
- For multistep solvers, state tensors are normal runtime tensors or
  session-owned buffers: model-output history, derivative history, last sample,
  lower-order counters, and corrector state.
- First stochastic parity should use runtime-provided noise tensors. RNG provider
  integration is a later design.
- Dynamic thresholding needs reduction support and should not be bundled into
  first pointwise scheduler kernels.
- Masked inpainting schedulers should fuse only the pointwise mask combine after
  original-image/noise tensors are explicit.

## 8. Validation plan

For each admitted scheduler slice:

- Generate schedule tables from the inspected Diffusers class and compare
  `timesteps`, `sigmas`, terminal sigma, and `init_noise_sigma`.
- Serialize the exact per-step coefficient pack that the CUDA kernel will
  receive, and compare it against Diffusers-derived scalar values.
- Run one-step parity with random latents and fixed model output.
- Run `add_noise` parity for img2img/inpaint/video2video schedule entry points.
- For stochastic branches, pass the same explicit noise tensor to Diffusers and
  DinoML instead of testing RNG parity.
- For multistep schedulers, test each warmup/history length separately before a
  full loop.
- Suggested first tolerances: fp32 scheduler-only `rtol=1e-5, atol=1e-6`;
  fp16/bf16 scheduler-only `rtol=2e-2, atol=2e-2` until fused kernels establish
  narrower model-specific bounds.
